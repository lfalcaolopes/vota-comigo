# Modelagem do vínculo votação ↔ proposição

A relação canônica entre votação e proposição é estabelecida exclusivamente pelo CSV `votacoesProposicoes-{ano}.csv`, que lista as proposições afetadas por cada votação. O arquivo `votacoesObjetos-{ano}.csv` e o campo `proposicaoObjeto` do JSON da API não são usados como vínculo canônico.

A modelagem é N:N: uma votação pode afetar mais de uma proposição, e uma proposição pode ter várias votações ao longo da tramitação. Quando uma votação afeta várias proposições, ela aparece associada a cada uma delas, sem regra especial de desempate ou escolha de uma proposição principal. A análise empírica de 2025 indicou que os casos com múltiplas proposições afetadas são minoritários, mas a estrutura precisa representá-los sem perda de informação.

`votacoesObjetos` foi descartado porque lista possíveis objetos da votação, acumulando proposições derivadas ao longo da tramitação. Na prática, funciona como log de objetos possíveis, não como chave confiável para a proposição que o produto deve exibir. O campo `proposicaoObjeto` do JSON da API também foi descartado como vínculo canônico porque aparece preenchido apenas em casos raros, normalmente quando uma peça acessória virou proposição própria cadastrada.

A identidade da peça efetivamente votada é uma informação textual da votação, não um identificador estruturado confiável. Para exibição e contexto, a ingestão complementa o esqueleto vindo dos CSVs com o JSON do endpoint `/votacoes/{id}` e preserva, em ordem de utilidade, `descUltimaAberturaVotacao`, `descUltimaApresentacaoProposicao` e `descricao`. Esses campos permitem distinguir texto-base, destaque, emenda aglutinativa, redação final e requerimentos procedurais quando a própria Câmara descreve a votação. Classificação de tipo regimental por regex sobre `descricao` ou outros textos foi rejeitada.
